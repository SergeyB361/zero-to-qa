# Неделя 4, День 5 — ООП на практике

## Зачем это нужно в QA
Классы помогают группировать данные и поведение, чтобы код тестов и утилит был понятнее.

## Где это полезно
- модели пользователя, заказа, окружения
- helper-классы для API и UI
- представление тест-кейсов и баг-репортов
- page objects и API clients

## Когда класс действительно нужен
Класс уместен, если объект:
- хранит состояние
- имеет связанные действия
- должен иметь понятный публичный API

Если тебе нужна одна маленькая функция без состояния, класс может быть лишним.

## Идея проектирования
Перед созданием класса ответь на три вопроса:
1. Какие данные хранит объект?
2. Какие действия объект умеет делать?
3. Что должно быть публичным API?

## Пример
```python
class TestCase:
    def __init__(self, title, status="draft"):
        self.title = title
        self.status = status

    def mark_passed(self):
        self.status = "passed"
```

Здесь:
- `title` и `status` — данные объекта
- `mark_passed()` — действие объекта

## Как не переусложнять
Плохой сигнал, если класс:
- ничего не хранит
- содержит одну функцию без связи с состоянием
- существует только "потому что так по-ООП"

Лучше простой класс с ясной ответственностью, чем красивая, но перегруженная архитектура.

## Практические правила
- Один класс — одна понятная роль.
- Начинай с простого интерфейса.
- Имена классов должны отражать сущность: `User`, `ApiClient`, `TestCase`.
- Не пытайся строить сложную иерархию, если можно обойтись парой простых классов.
