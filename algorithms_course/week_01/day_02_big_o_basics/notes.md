# Неделя 1, День 2 - Сложность: Big O

## Зачем нужна оценка сложности
Одну и ту же задачу можно решить по-разному.
Важно понимать не только то, что решение работает, но и то, как его стоимость растёт при увеличении входных данных.

Для этого используют оценку сложности, чаще всего в формате Big O.

## Что показывает Big O
Big O показывает, как растёт число операций при росте размера входа `n`.

Это не точное время в секундах.
Это модель роста.

## Основные классы сложности
- `O(1)` - константное время
- `O(log n)` - логарифмическое
- `O(n)` - линейное
- `O(n log n)` - типично для хороших сортировок
- `O(n^2)` - часто вложенные циклы

## Примеры
### `O(1)`
Взять первый элемент списка.

### `O(n)`
Пройти по всему списку один раз.

### `O(n^2)`
Сравнивать каждый элемент с каждым.

### `O(log n)`
Бинарный поиск в отсортированном массиве.

## Как прикидывать сложность
Смотри на структуру решения:
- один цикл по `n` -> обычно `O(n)`;
- два вложенных цикла по `n` -> обычно `O(n^2)`;
- деление диапазона пополам -> часто `O(log n)`.

## Что не так важно на старте
В Big O обычно отбрасывают:
- константы;
- менее значимые слагаемые.

Например:
- `O(2n)` -> `O(n)`
- `O(n + 100)` -> `O(n)`
- `O(n^2 + n)` -> `O(n^2)`

## Время и память
Есть два ресурса:
- время работы;
- память.

Иногда решение быстрее, но требует больше памяти.
Иногда наоборот.

## Типичные ошибки
1. Считать количество строк кода вместо операций.
2. Игнорировать вложенность циклов.
3. Считать, что `O(n)` и `O(log n)` отличаются мало.
4. Не учитывать предварительные условия, например отсортированность.

## Практическое правило
Когда видишь решение, задай себе вопросы:
1. Сколько раз мы проходим по данным?
2. Есть ли вложенные циклы?
3. Есть ли структура данных, которая ускоряет доступ?
