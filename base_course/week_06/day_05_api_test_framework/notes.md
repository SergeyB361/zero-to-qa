# Неделя 6, День 5 — API тест-фреймворк

## Зачем нужна структура
Если тестов становится больше, неудобно держать всё в одном файле. Нужна структура: где хранить clients, endpoints, helpers, config.

## Простая идея фреймворка
- `tests/` — тесты
- `clients/` — работа с API
- `utils/` — вспомогательные функции
- `config/` — настройки

## Почему это полезно
- меньше дублирования
- удобнее переиспользовать запросы
- проще менять base URL, headers и общие настройки

## Пример роли client-класса
API client может брать на себя:
- отправку запросов
- базовый URL
- общие headers
- часто используемые методы API

## Практические правила
- не смешивай тесты и низкоуровневую логику запросов в одном месте
- тест должен описывать сценарий, а не сборку URL по кускам
- структура должна быть полезной, а не слишком сложной для текущего объёма проекта
