# Pet-проект для портфолио Junior Python Developer

## Название проекта
**Task Service + Audit Timeline**

## Формат проекта
Pet-проект для портфолио Junior Python Developer.

Проект состоит из **двух небольших backend-сервисов**:
- `Task Service` — управляет задачами
- `Audit Timeline Service` — принимает события и хранит историю изменений

## Главная цель
Сделать проект, который:
- выглядит сильнее обычного CRUD;
- остаётся реалистичным для Junior-разработчика;
- имеет MVP, который можно реализовать примерно за **10 рабочих дней**;
- удобно показывать в GitHub, README и на собеседовании.

## Ключевое ограничение MVP
MVP должен быть **простым, законченным и демонстрируемым**, а не "архитектурно модным, но недоделанным".

Поэтому в первой версии проект **не должен пытаться быть production-grade системой**.

---

# MVP-версия проекта

## Что именно мы делаем
В MVP реализуются **два локальных FastAPI-сервиса**, которые взаимодействуют между собой по обычному HTTP.

### Сервис 1 — Task Service
Отвечает за бизнес-операции с задачами:
- создать задачу;
- получить задачу;
- получить список задач;
- изменить статус;
- изменить исполнителя;
- удалить задачу.

После каждого бизнес-действия сервис отправляет событие в Audit Timeline Service.

### Сервис 2 — Audit Timeline Service
Отвечает за аудит:
- принимает события;
- сохраняет их в append-only журнал;
- возвращает timeline по задаче;
- возвращает timeline по пользователю-инициатору;
- строит snapshot текущего состояния задачи по событиям.

# Архитектурное решение для MVP

## Обязательное решение
Чтобы проект не расползался по сложности, для MVP фиксируется такая архитектура:

- `Task Service` — FastAPI + SQLite
- `Audit Timeline Service` — FastAPI + SQLite
- взаимодействие сервисов — **синхронный HTTP POST**
- запуск локально на двух портах
- данные хранятся локально

## Что это значит practically
После успешного действия в `Task Service`:
1. изменяются данные задачи;
2. создаётся событие;
3. событие отправляется HTTP-запросом в `Audit Timeline Service`;
4. audit-сервис сохраняет событие.

## Что делать при ошибке отправки события в MVP
Чтобы MVP остался подъёмным, принимается простое правило:
- если отправка события в audit-сервис не удалась, `Task Service` возвращает ошибку клиенту;
- сложные retry-механизмы и outbox не входят в первую версию.

Это упрощение осознанное и допустимое для Junior MVP.

# Предметная область

Проект моделирует простую task-систему.

Есть задача, у которой во времени меняются поля и состояние.

Важно хранить не только текущее состояние задачи, но и историю действий:
- кто создал задачу;
- кто сменил статус;
- кто назначил исполнителя;
- кто удалил задачу.

- `title`
- `description`
- `status`
- `assignee_id`
- `created_by`
- `created_at`
- `updated_at`
- `is_deleted`

## Domain Event
Минимальные поля:

- `event_type`
- `entity_type`
- `entity_id`
- `actor_id`
- `occurred_at`
- `payload`

---

# Что считается пользователем в MVP

Чтобы убрать двусмысленность, в MVP фиксируются два разных смысла user-id:

- `actor_id` — пользователь, который **совершил действие**;
- `assignee_id` — пользователь, который **назначен исполнителем задачи**.

## Важно
Timeline по пользователю в MVP строится **по `actor_id`**, то есть по инициатору действия.

Не по исполнителю задачи.

# События в MVP

## Обязательные типы событий
- `task.created`
- `task.status_changed`
- `task.assignee_changed`
- `task.deleted`

## Что убираем из MVP
Чтобы уложиться в 10 рабочих дней, из первой версии исключаем:

Их можно добавить во вторую итерацию.

## Payload по событиям
### `task.created`
Содержит начальное состояние задачи:
- `title`
- `description`
- `status`
- `assignee_id`
- `created_by`

### `task.status_changed`
Содержит:
- `old_status`
- `new_status`

### `task.assignee_changed`
Содержит:
- `old_assignee_id`
- `new_assignee_id`

### `task.deleted`
Содержит:
- `is_deleted = true`

# API-контракт MVP

## Task Service
Базовый URL:
- `/tasks`

### Обязательные endpoint'ы
- `POST /tasks` — создать задачу
- `GET /tasks` — получить список задач
- `GET /tasks/{task_id}` — получить задачу
- `PATCH /tasks/{task_id}/status` — изменить статус
- `PATCH /tasks/{task_id}/assignee` — изменить исполнителя
- `DELETE /tasks/{task_id}` — удалить задачу

## Audit Timeline Service
Базовый URL:
- `/events`
- `/timeline`
- `/snapshot`

### Обязательные endpoint'ы
- `POST /events` — принять событие
- `GET /events` — получить список событий
- `GET /events/{event_id}` — получить событие
- `GET /timeline/tasks/{task_id}` — timeline по задаче
- `GET /timeline/users/{actor_id}` — timeline по пользователю-инициатору
- `GET /snapshot/tasks/{task_id}` — snapshot текущего состояния задачи

---

# Как должен работать snapshot в MVP

Чтобы не было двусмысленности, фиксируем простое правило.

## Snapshot задачи
Snapshot — это текущее состояние задачи, восстановленное **только из событий этой задачи**.

Он должен включать:
- `task_id`
- `title`
- `description`
- `status`
- `assignee_id`
- `created_by`
- `is_deleted`
- `last_event_at`

## Как трактовать удалённую задачу
Если задача удалена:
- snapshot всё равно должен возвращаться;
- в нём должно быть `is_deleted = true`.

То есть удаление не стирает историю и не делает snapshot невозможным.

# План реализации по этапам

Один этап может занимать несколько рабочих дней.

Цель этого плана — держать MVP в реальном Junior-scope и не распыляться.

## Этап 1 — Подготовка каркаса проекта
**Оценка:** `0.5–1 день`

Что входит:
- создать репозиторий и структуру папок;
- поднять два FastAPI-сервиса;
- сделать health endpoint'ы;
- проверить локальный запуск обоих сервисов.

Результат этапа:
- оба сервиса запускаются локально;
- есть понятная структура проекта.

## Этап 2 — Базовый Task Service
**Оценка:** `2–3 дня`

Что входит:
- реализовать модель `Task`;
- настроить SQLite;
- сделать `POST /tasks`;
- сделать `GET /tasks/{task_id}`;
- сделать `GET /tasks`;
- реализовать `PATCH /tasks/{task_id}/status`;
- реализовать `PATCH /tasks/{task_id}/assignee`;
- реализовать `DELETE /tasks/{task_id}`.

Результат этапа:
- Task Service выполняет все базовые операции MVP.

## Этап 3 — Audit Timeline Service
**Оценка:** `1.5–2.5 дня`

Что входит:
- реализовать модель события;
- настроить SQLite для audit-сервиса;
- сделать `POST /events`;
- сделать `GET /events`;
- сделать `GET /events/{event_id}`;
- обеспечить append-only сохранение событий.

Результат этапа:
- audit-сервис принимает и хранит события.

## Этап 4 — Интеграция сервисов
**Оценка:** `1–2 дня`

Что входит:
- после действия в Task Service формировать доменное событие;
- отправлять событие по HTTP в Audit Timeline Service;
- обработать базовый сценарий ошибки отправки;
- проверить цепочку `действие -> событие -> сохранение`.

Результат этапа:
- изменение задачи приводит к появлению события в audit-сервисе.

**Оценка:** `1.5–2.5 дня`

Что входит:

- реализовать `GET /snapshot/tasks/{task_id}`;
- проверить восстановление текущего состояния задачи по событиям.

Результат этапа:
- можно показать историю изменений и итоговое состояние задачи.

## Этап 6 — Тесты, README и demo flow
**Оценка:** `1.5–2 дня`

Что входит:
- написать базовые тесты;
- описать запуск проекта;
- оформить README;
- подготовить короткий демонстрационный сценарий;

Результат этапа:

## Общая оценка по сроку
- минимально: около `8` рабочих дней
- реалистично: около `10` рабочих дней
- с небольшим запасом на исправления: до `12` рабочих дней

## Оценка при темпе 1.5 часа в день
Если работать над проектом в среднем по `1.5 часа в день`, то оценка по этапам будет такой:

- Этап 1 — `3–5` дней
- Этап 2 — `11–16` дней
- Этап 3 — `8–13` дней
- Этап 4 — `5–11` дней
- Этап 5 — `8–13` дней
- Этап 6 — `8–11` дней

### Итог при таком ритме
- общая трудоёмкость MVP: примерно `64–96` часов
- если заниматься почти каждый день: примерно `1.5–2 месяца`
- если заниматься `5` дней в неделю: примерно `9–13 недель`

### Практический ориентир
Для спокойного прохождения без спешки разумно закладывать `около 2 месяцев`.
Если проект идёт параллельно с основным курсом и будут паузы, лучше ориентироваться на `2–3 месяца`.

# Итоговая формулировка задания

Разработать portfolio pet-project для Junior Python Developer, состоящий из двух FastAPI-сервисов:
- `Task Service`

`Task Service` управляет задачами и после каждого действия отправляет доменное событие в `Audit Timeline Service`.

`Audit Timeline Service` хранит append-only журнал событий и предоставляет:
- timeline по задаче;
- timeline по пользователю-инициатору;
- snapshot текущего состояния задачи.

Проект должен быть реалистичным, законченным, локально воспроизводимым и реализуемым как MVP примерно за 10 рабочих дней Junior-разработчиком.
