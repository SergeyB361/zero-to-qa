# Согласование pet project с обучающим курсом

## Зачем нужен этот файл
Этот документ показывает:
- какие знания для pet project уже покрыты основным курсом;
- какие знания появятся позже;
- когда каждый этап проекта реально лучше запускать;
- где сейчас есть разрыв между проектом и учебной программой.

## Короткий вывод
Pet project хорошо согласуется с курсом по общей траектории развития навыков, но не полностью совпадает с текущей точкой обучения.

Итоговая оценка:
- по логике развития навыков: `9/10`
- по готовности начать полный MVP прямо сейчас: `5/10`
- по готовности начать подготовительный этап уже сейчас: `8/10`

## Что уже покрыто курсом
На текущем этапе основного курса уже изучены темы, которые помогают понимать и проектировать pet project:

- базовый Python;
- структуры данных;
- файлы и JSON;
- type hints;
- logging;
- pathlib;
- базовое ООП: начало `Week 4`.

Это уже позволяет:
- описывать сущности `Task` и `Domain Event`;
- продумывать payload событий;
- строить структуру проекта;
- писать часть бизнес-логики без веб-слоя;
- понимать MVP-границы и демонстрационный сценарий.

## Что pet project требует, но курс ещё не довёл
Для полного MVP из [spec.md](spec.md) нужны темы, которые либо ещё впереди, либо только начинаются:

- методы, наследование, инкапсуляция, ООП-практика: `Week 4`;
- основы тестирования и pytest: `Week 5`;
- HTTP, REST, requests, API-подход: `Week 6`;
- SQLite: `Week 8`;
- архитектура проекта и полировка портфолио: `Week 10`.

### Этап 5 — Timeline и Snapshot

## Практический вывод по старту проекта
Сейчас не стоит пытаться параллельно тащить весь MVP.

Правильная схема такая:

1. Уже сейчас
- держать ТЗ;
- держать roadmap;
- уточнять доменную модель;
- фиксировать структуру проекта;
- при желании сделать очень простой in-memory набросок без полного API.

2. После `Week 4`
- уверенно собрать модели и часть бизнес-логики.

3. После `Week 5-6`
- начинать реальный MVP с API и межсервисным взаимодействием.

4. После `Week 8`
- довести хранилище и timeline/snapshot более уверенно.

## Рекомендация
Pet project не противоречит курсу. Наоборот, он хорошо подходит как сквозная практическая цель.

Но лучший режим работы такой:
- сейчас: подготовка и проектирование;
- после `Week 6`: старт основной реализации;
- после `Week 8`: доведение до полноценного MVP;
- после `Week 10`: упаковка под портфолио.

## Что считать нормальным темпом
Для текущего уровня нормально:
- не спешить с полной реализацией сервисов;
- сначала строить понимание предметной области;
- запускать большие технические этапы только после того, как курс подведёт нужную теорию.

Это не замедление, а нормальная синхронизация сложности проекта с ростом навыков.
