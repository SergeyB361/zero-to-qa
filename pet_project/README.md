# Task Service + Audit Timeline

Pet project для портфолио Junior Python Developer.

## Идея проекта
Проект состоит из двух небольших backend-сервисов:
- `Task Service` — управляет задачами;
- `Audit Timeline Service` — принимает доменные события и хранит историю изменений.

Цель проекта — показать не просто CRUD, а понятный бизнес-поток:

`действие с задачей -> доменное событие -> аудит -> timeline -> snapshot`

## Почему проект сильный для портфолио
Проект демонстрирует:
- FastAPI;
- REST API;
- работу с SQLite;
- разбиение системы на два сервиса;
- межсервисное HTTP-взаимодействие;
- доменные события;
- аудит изменений;
- восстановление текущего состояния из истории событий.

## MVP-границы
Первая версия проекта специально упрощена.

В MVP входят:
- создание задачи;
- получение задачи;
- получение списка задач;
- смена статуса;
- смена исполнителя;
- удаление задачи;
- генерация событий;
- приём и хранение событий;
- timeline по задаче;
- timeline по пользователю-инициатору;
- snapshot текущего состояния задачи.

Не входят в MVP:
- комментарии;
- изменение заголовка;
- сложная авторизация;
- retry / outbox / брокеры сообщений;
- Docker как обязательная часть;
- сложная production-инфраструктура.

## Архитектура MVP
Фиксированное решение для первой версии:
- `Task Service` — FastAPI + SQLite;
- `Audit Timeline Service` — FastAPI + SQLite;
- взаимодействие между сервисами — синхронный HTTP POST.

### Схема взаимодействия
```text
Client
  |
  v
Task Service
  |
  | POST domain event
  v
Audit Timeline Service
```

## Демонстрационный сценарий MVP
Проект должен позволять быстро показать такой flow:
1. создать задачу;
2. сменить статус;
3. сменить исполнителя;
4. удалить задачу;
5. открыть timeline по задаче;
6. открыть timeline по пользователю;
7. открыть snapshot и подтвердить итоговое состояние.

## Принцип реализации
- сначала делается законченный MVP;
- всё, что не входит в MVP, не раздувает первую версию;
- важнее завершённость и демонстрационная пригодность, чем лишняя архитектурная сложность.
