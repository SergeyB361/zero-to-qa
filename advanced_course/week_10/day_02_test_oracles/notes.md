# Quality Engineering и системное мышление, День 2 — Test oracles

## Зачем это нужно
- Тест без понятного oracle не знает, что считать правильным результатом.
- Хороший oracle делает expected result защищаемым, а не интуитивным.
- Для зрелого QA это одна из ключевых тем качества самих проверок.

## Что важно понять
- Oracle может приходить из requirements, domain rules, contract, UX expectations или предыдущего устойчивого состояния.
- Один и тот же тест может иметь несколько источников oracle.
- Если oracle слабый, тест легко становится ложным или бессмысленным.

## Частые ошибки
- Опираться только на интуицию.
- Путать поведение текущей реализации с правильным поведением.
- Не фиксировать, откуда взят expected result.
