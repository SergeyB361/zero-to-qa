# Bonus Advanced Python OOP, День 2 - __eq__ и __hash__

## Зачем это нужно

По умолчанию Python сравнивает объекты по идентичности в памяти.

То есть два разных объекта с одинаковыми данными считаются разными:

```python
class UserId:
    def __init__(self, value):
        self.value = value

u1 = UserId("user-1")
u2 = UserId("user-1")

print(u1 == u2)  # False
```

Для прикладных моделей это часто неверно.

Если у тебя есть:
- `TaskId("TASK-101")`
- `UserId("user-42")`
- `EventKey("task.created", "TASK-101")`

то логически такие объекты обычно сравнивают по значению, а не по адресу в памяти.

Именно для этого нужны `__eq__` и `__hash__`.

## Что делает __eq__

`__eq__` отвечает за сравнение через `==`.

Если ты реализуешь `__eq__`, то задаёшь правило:
- когда два объекта считаются равными;
- по каким полям их нужно сравнивать.

Пример:

```python
class UserId:
    def __init__(self, value):
        self.value = value

    def __eq__(self, other):
        if not isinstance(other, UserId):
            return NotImplemented
        return self.value == other.value
```

Теперь:

```python
UserId("user-1") == UserId("user-1")  # True
```

## Что делает __hash__

`__hash__` нужен для использования объекта в:
- `set`;
- ключах `dict`;
- любых hash-based структурах.

Пример:

```python
tags = {Tag("api"), Tag("api")}
```

Если `Tag("api")` и `Tag("api")` должны считаться одним и тем же значением, то обычно нужно определить и `__eq__`, и `__hash__`.

## Главный контракт

Самое важное правило:

- если `a == b`, то `hash(a)` обязан быть равен `hash(b)`.

Иначе поведение `set` и `dict` станет некорректным.

Пример плохой модели:
- объекты сравниваются по `id`;
- а хэш считаешь по `title`.

Так делать нельзя.

## Когда достаточно только __eq__

Если объект не будет использоваться в `set` и `dict`, иногда достаточно только `__eq__`.

Например:
- сравнение DTO в тесте;
- value-object в обычных assertions;
- сравнение двух событий в списке.

Но если объект должен быть ключом словаря или элементом множества, нужен и `__hash__`.

## Что делает Python, если определить __eq__, но не определить __hash__

Обычно такой объект становится unhashable.

Это защитный механизм Python.

Он не даёт использовать в `set` и `dict` объект, для которого equality уже переопределён, а корректный hash ещё не описан.

## Когда hash делать нельзя

Если объект изменяемый и поля, участвующие в сравнении, могут меняться, делать `__hash__` опасно.

Почему:
- объект попал в `set`;
- потом изменилось поле, от которого зависит hash;
- структура больше не может корректно найти этот объект.

Поэтому хорошее правило:
- hashable-объекты должны быть логически неизменяемыми.

## Хорошие кандидаты для __eq__ и __hash__

Обычно это:
- идентификаторы;
- value objects;
- ключи событий;
- структуры, которые удобно хранить в `set` или использовать как ключи `dict`.

Примеры:
- `TaskId`
- `UserId`
- `EventKey`
- `Money`
- `Coordinate`

## Как писать __eq__ аккуратно

Обычно логика такая:

```python
def __eq__(self, other):
    if not isinstance(other, MyClass):
        return NotImplemented
    return (self.field1, self.field2) == (other.field1, other.field2)
```

Почему `NotImplemented`, а не `False`:
- это корректный способ сообщить Python, что сравнение с другим типом не поддерживается;
- Python сам попробует альтернативную логику.

## Как писать __hash__ аккуратно

Частый паттерн:

```python
def __hash__(self):
    return hash((self.field1, self.field2))
```

То есть хэш считается по кортежу стабильных полей.

## Практический смысл для QA и backend

### 1. Сравнение доменных моделей в тестах

Если у объектов есть понятный `__eq__`, писать assertions проще:

```python
assert actual_event == expected_event
```

вместо ручного сравнения каждого поля.

### 2. Дедупликация через set

Можно убирать дубликаты по смыслу, а не по объектной идентичности.

### 3. Ключи словарей

Удобно хранить:
- агрегаты по `EventKey`;
- кэш по `UserId`;
- счётчики по `TaskId`.

## Типичные ошибки

### 1. __eq__ есть, __hash__ нет

Если объект должен жить в `set`/`dict`, это ошибка проектирования.

### 2. hash считается не по тем же полям, что equality

Это ломает контракт.

### 3. hash у мутабельного объекта

Это одна из самых неприятных ошибок в доменных моделях.

### 4. Сравнение только части значимых полей

Тогда equality будет логически неверным.

## Практическое правило для курса

Если класс описывает value object, сначала задай себе 3 вопроса:

1. По каким полям объекты считаются равными?
2. Будет ли этот объект использоваться в `set` или как ключ `dict`?
3. Эти поля стабильны или могут меняться?

Если ответы такие:
- равенство по значению;
- нужен `set`/`dict`;
- поля стабильны,

то обычно нужен и `__eq__`, и `__hash__`.
