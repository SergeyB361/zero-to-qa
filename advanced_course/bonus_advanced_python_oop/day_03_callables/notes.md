# Bonus Advanced Python OOP, День 3 - __call__ и callable objects

## Зачем это нужно

В Python функцию можно вызвать так:

```python
result = func(value)
```

Но вызывать можно не только функции. Если объект реализует `__call__`, он тоже становится вызываемым:

```python
result = obj(value)
```

Такие объекты называют **callable objects**.

Это полезно, когда ты хочешь хранить внутри объекта:
- состояние;
- конфигурацию;
- параметры проверки;
- накопленную статистику;
- несколько связанных правил.

## Что делает __call__

`__call__` определяет поведение объекта при вызове через круглые скобки.

Пример:

```python
class Doubler:
    def __call__(self, value):
        return value * 2
```

Тогда:

```python
doubler = Doubler()
print(doubler(10))  # 20
```

## Чем callable object отличается от обычной функции

Функция хороша, когда:
- логика короткая;
- не нужно хранить состояние;
- нет смысла делать отдельный класс.

Callable object хорош, когда:
- у поведения есть настройки;
- нужно помнить предыдущие вызовы;
- удобно инкапсулировать логику в отдельный объект;
- один и тот же объект должен вести себя как “настроенная функция”.

## Где это реально полезно

### 1. Валидаторы

Например, объект хранит минимальную длину строки:

```python
validator = MinLengthValidator(8)
validator("secret123")
```

### 2. Фильтры

Можно создать объект, который проверяет событие или запись лога по набору правил.

### 3. Мапперы / преобразователи

Callable object удобно использовать для нормализации данных.

### 4. Сборщики статистики

Объект можно вызывать как функцию и одновременно считать количество вызовов.

## callable() и __call__

В Python есть встроенная функция `callable()`:

```python
callable(obj)
```

Она отвечает на вопрос: можно ли вызвать объект как функцию.

Пример:

```python
callable(len)         # True
callable("hello")     # False
callable(validator)   # True
```

## Пример с состоянием

```python
class RequestCounter:
    def __init__(self):
        self.calls = 0

    def __call__(self):
        self.calls += 1
        return self.calls
```

Теперь один и тот же объект:
- ведёт себя как функция;
- хранит внутреннее состояние.

## Когда __call__ уместен

Используй `__call__`, если объект действительно воспринимается как действие.

Хорошие примеры:
- валидатор;
- предикат;
- фильтр;
- фабрика событий;
- преобразователь данных.

Плохие примеры:
- обычная доменная сущность, которую просто хочется “сделать поумнее”;
- класс, где явный метод `run()` или `validate()` читается лучше.

Если `obj()` по смыслу неочевиден, `__call__` не нужен.

## Сравнение с обычным методом

Иногда эти варианты равносильны:

```python
validator.validate(data)
validator(data)
```

Второй вариант короче, но требует, чтобы читатель кода понимал, что объект callable.

Поэтому правило простое:
- если объект действительно выглядит как “функция с настройками”, `__call__` уместен;
- если нужна явность, лучше обычный метод.

## Практический смысл для QA и backend

### 1. Переиспользуемые проверки

Можно сделать объект-проверку:

```python
is_high_priority = PriorityFilter("high")
```

И затем использовать его в нескольких местах.

### 2. Настраиваемые валидаторы

Например:
- длина строки;
- допустимые статусы;
- формат идентификатора;
- проверка payload события.

### 3. Тестовые утилиты

Callable object удобно использовать как:
- фабрику тестовых данных;
- matcher;
- predicate для фильтрации результатов.

## Типичные ошибки

### 1. Использовать __call__ там, где обычный метод лучше

Если по смыслу нужен `send()`, `validate()`, `match()`, `build()`, не надо насильно прятать всё в `__call__`.

### 2. Делать объект callable, но неочевидным

Если непонятно, что именно делает `obj()`, код становится хуже, а не лучше.

### 3. Прятать слишком сложную бизнес-логику

`__call__` не должен превращаться в “магический вход во всё подряд”.

## Практическое правило для курса

Хороший кандидат на `__call__` — это объект, который можно назвать так:
- validator
- filter
- matcher
- mapper
- formatter
- factory

Если название класса уже звучит как действие или инструмент, `__call__` обычно читается нормально.
