# Bonus Advanced Python OOP, День 1 - __str__ и __repr__

## Зачем это нужно

Когда ты печатаешь объект Python без специальных методов, результат обычно бесполезен:

```python
<__main__.Task object at 0x000001F3...>
```

Для реального проекта этого мало. В логах, отладке, тестах и ошибках важно быстро понимать:
- что это за объект;
- какие у него ключевые поля;
- чем один объект отличается от другого.

Именно для этого и нужны `__str__` и `__repr__`.

## Что делает __str__

`__str__` - это человекочитаемое представление объекта.

Обычно его используют, когда объект нужно показать:
- пользователю;
- в обычном `print()`;
- в коротком логе;
- в понятном сообщении об ошибке.

Пример:

```python
class Task:
    def __str__(self):
        return "TASK-101: Fix login bug"
```

Тогда:

```python
print(task)
```

даст:

```python
TASK-101: Fix login bug
```

## Что делает __repr__

`__repr__` - это представление объекта для разработчика.

Оно нужно для:
- отладки;
- работы в консоли Python;
- анализа списков и словарей с объектами;
- диагностики падений тестов.

Хороший `__repr__` обычно:
- более точный, чем `__str__`;
- показывает имя класса;
- показывает ключевые поля;
- не скрывает важный контекст.

Пример:

```python
class Task:
    def __repr__(self):
        return "Task(task_id='TASK-101', title='Fix login bug', status='open')"
```

## Как Python выбирает, что использовать

### `print(obj)`

Python вызывает `str(obj)`, а значит пытается использовать `obj.__str__()`.

### `repr(obj)`

Python вызывает `obj.__repr__()`.

### Списки, множества, словари

Когда объект лежит внутри коллекции, Python обычно использует `repr()`.

Пример:

```python
print([task])
```

Чаще всего покажет именно `__repr__`, а не `__str__`.

### F-строки

```python
f"{task}"
```

использует `str()`.

```python
f"{task!r}"
```

использует `repr()`.

## Что будет, если __str__ нет

Если `__str__` не определён, Python обычно использует `__repr__` как запасной вариант.

Это полезно помнить:
- можно начать только с `__repr__`;
- но в прикладных моделях часто лучше иметь оба метода.

## Практическое различие

Хорошее правило:

- `__str__` - коротко и понятно для человека;
- `__repr__` - точно и полезно для разработчика.

Пример для `Task`:

```python
__str__  -> "TASK-101: Fix login bug"
__repr__ -> "Task(task_id='TASK-101', title='Fix login bug', status='open')"
```

## Где это реально полезно в QA и backend

### 1. Логи

Если логируешь объект без `__str__`/`__repr__`, лог бесполезен.

С хорошим представлением объекта сразу видно:
- какой task обрабатывался;
- какое событие упало;
- какие данные были у запроса.

### 2. Падающие тесты

Когда `assert` падает, Python показывает представление объектов.

Если `__repr__` хороший, падение читается быстрее.

### 3. Отладка доменных моделей

В pet project с `Task`, `DomainEvent`, `TimelineEntry` эти методы особенно полезны.

## Когда достаточно только __repr__

Если объект технический и не нужен пользователю напрямую, иногда достаточно одного `__repr__`.

Это нормально для:
- внутренних DTO;
- событий;
- объектов инфраструктуры.

## Частые ошибки

### 1. Возвращать не строку

Оба метода обязаны возвращать `str`.

Неправильно:

```python
def __repr__(self):
    return 123
```

### 2. Пихать слишком много данных

Не надо выводить весь payload, большие списки, секреты или токены.

Плохой `__repr__`:
- шумный;
- медленный;
- иногда опасный по безопасности.

### 3. Делать __repr__ слишком красивым, но бесполезным

Если из строки непонятно, что это за класс и какие у него ключевые поля, `__repr__` теряет смысл.

## Практическое правило для курса

Для большинства твоих моделей ориентируйся так:

- `__str__`:
  - короткий читаемый текст;
  - пригоден для `print()`.

- `__repr__`:
  - имя класса;
  - важные поля;
  - без лишнего шума.
