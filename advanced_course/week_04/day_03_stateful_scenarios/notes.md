# API: reliability и integration, День 3 — Stateful scenarios

## Зачем это нужно
Многие API зависят от текущего состояния сущности: заказ нельзя refund-ить до оплаты, тикет нельзя re-open без закрытия, пользователь не может активироваться дважды одинаково.

## Что важно понять
Stateful test проверяет не один isolated запрос, а переходы между состояниями.

Нужно явно определить:
- allowed transitions
- forbidden transitions
- terminal states
- side effects каждого перехода

## Где чаще всего ошибки
- система разрешает недопустимый переход
- код ответа не соответствует business conflict
- объект меняется частично и остаётся в неконсистентном состоянии

## Как строить тест
Полезная последовательность:
1. зафиксировать initial state
2. выполнить переход
3. проверить новый state
4. попробовать запрещённый переход
5. убедиться, что state не испортился

## Что запомнить
- stateful scenario всегда проверяет переход, а не только endpoint
- forbidden transition должен иметь понятную ошибку
- side effect после неудачного перехода надо проверять отдельно
- allowed и disallowed path должны быть явными
