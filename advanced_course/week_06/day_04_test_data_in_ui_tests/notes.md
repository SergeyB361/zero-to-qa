# UI: качество и масштабирование, День 4 — Test data in UI tests

## Зачем это нужно
- UI-тесты часто падают не из-за UI, а из-за грязных или конфликтующих данных.
- Управление данными в UI-слое должно быть столь же дисциплинированным, как и в API.
- Изоляция данных напрямую влияет на параллельный запуск.

## Что важно понять
- Нужны уникальные сущности, предсказуемые роли и controlled cleanup.
- Часть данных лучше создавать через API, а не через UI.
- Reuse грязного пользователя между тестами быстро создаёт flaky.

## Частые ошибки
- Один shared user на все UI-тесты.
- Создание данных только через UI там, где быстрее API.
- Отсутствие cleanup или уникальных идентификаторов.
