# ТЗ: Мини-проект — устойчивый PyTest suite

## Исходный код
Используй файл:
- `qa_service.py`

В нём есть четыре зоны для тестирования:
1. `calculate_discount()` — pure logic, хорошо подходит под параметризацию
2. `register_user()` — есть внешние зависимости `repo` и `mailer`, здесь нужен Mock
3. `write_audit_report()` — работа с файловой системой, здесь нужен `tmp_path`
4. `build_invite_code()` — зависимость от генератора, важно сделать тест детерминированным

## Что нужно сделать
Создай устойчивый набор тестов, который покрывает:
- позитивные сценарии
- негативные сценарии
- граничные значения
- корректность вызова внешних зависимостей

## Обязательные требования
1. Использовать минимум одну composition fixture.
2. Использовать минимум одну factory fixture.
3. Использовать параметризацию с читаемыми `ids`.
4. Использовать хотя бы один `smoke` marker.
5. Использовать Mock для `repo` и/или `mailer`.
6. Для файлов использовать `tmp_path`, а не общий путь в проекте.
7. Тесты не должны зависеть от порядка запуска.
8. Тесты должны быть совместимы с `pytest -n auto`.

## Что не нужно делать
- реальные HTTP-запросы
- реальные письма
- общие временные файлы в репозитории
- `sleep` как способ стабилизации
- retry вместо поиска причины

## Рекомендуемая структура
- `test_discounts.py`
- `test_registration.py`
- `test_reports.py`
- `conftest.py`
- при желании `pytest.ini`

## Критерии качества
Хорошее решение:
- короткие и понятные тесты
- осмысленные names и ids
- проверка результата и side effects
- отсутствие flaky-паттернов
- понятная структура suite

## Команды проверки
- `pytest -v`
- `pytest -m smoke`
- `pytest -n auto`
