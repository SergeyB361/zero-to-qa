# PyTest: продвинутый уровень, День 5 — Flaky tests

## Что такое flaky test
Это тест, который то проходит, то падает без изменения целевого кода.

Для команды это одна из самых дорогих проблем:
- подрывает доверие к автотестам
- замедляет релизы
- маскирует реальные дефекты

## Частые причины
- shared state между тестами
- зависимость от времени
- случайность (`random`)
- сеть и внешние сервисы
- хрупкие ожидания в UI
- порядок запуска тестов
- общие файлы и общая БД

## Как думать правильно
Flaky test обычно не «лечат ретраями» в первую очередь.
Сначала ищут источник нестабильности.

Полезная последовательность:
1. воспроизвести
2. понять зависимость
3. изолировать состояние
4. сделать входы детерминированными
5. только потом думать о retry

## Что помогает
- `tmp_path` вместо общих файлов
- фиктивные данные вместо случайных
- dependency injection для времени и random
- независимые тесты
- явные условия ожидания вместо `sleep`

## Что почти всегда плохая идея
- `time.sleep()` как универсальное решение
- глобальные фикстуры с изменяемым состоянием
- запись в один и тот же файл из разных тестов
- тесты, зависящие от порядка запуска

## Что собирать для диагностики
- traceback
- логи
- входные данные
- время запуска
- окружение
- артефакты конкретного теста
