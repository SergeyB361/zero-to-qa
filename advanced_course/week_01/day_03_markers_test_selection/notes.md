# PyTest: продвинутый уровень, День 3 — Markers и test selection

## Зачем это нужно
Когда тестов много, важна не только их корректность, но и управляемость:
- что запустить быстро
- что пойдёт в smoke
- что исключить временно
- как отделить API от UI и unit от integration

Для этого нужны markers и селекция запуска.

## Практически полезные маркеры
Часто достаточно небольшого набора:
- `smoke`
- `regression`
- `api`
- `ui`
- `slow`
- `integration`

Важно не плодить десятки маркеров без системы.

## Что делают основные mark-инструменты
`@pytest.mark.<name>`:
- помечает тест категорией

`@pytest.mark.skip`:
- тест не запускается совсем

`@pytest.mark.xfail`:
- ожидаемая проблема
- полезно для известного бага
- не должно превращаться в кладбище нерешённых проблем

## Selection
Полезные команды:
- `pytest -m smoke`
- `pytest -m "api and not slow"`
- `pytest -k login`

`-m` работает по marker.
`-k` ищет по имени теста и подстрокам.

## Регистрация custom markers
Если marker свой, его лучше зарегистрировать в `pytest.ini`.
Иначе проект быстро превращается в набор случайных тегов.

Пример идеи:
- `smoke: critical fast checks`
- `slow: long-running tests`
- `api: api test layer`

## Antipatterns
Плохо:
- помечать почти все тесты как `smoke`
- использовать `xfail` вместо фикса бага
- хранить в `skip` пол-проекта месяцами
- создавать marker под каждый второй файл
