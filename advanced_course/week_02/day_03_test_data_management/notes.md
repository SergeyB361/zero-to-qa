# Test Data Engineering, День 3 — Test data management

## Зачем это нужно
Даже хорошие factories не спасают, если данные в проекте лежат хаотично.

Test data management отвечает на практические вопросы:
- какие данные reusable
- какие генерируются на лету
- что можно шарить между тестами, а что нельзя
- как не потерять контроль над валидностью наборов

## Что важно понять
Обычно данные удобно делить на слои:
- baseline datasets: стабильные и переиспользуемые данные
- dynamic data: данные под конкретный запуск
- edge datasets: границы, невалидные и редкие случаи

Ключевая цель — не хранить всё в одном giant JSON и не размазывать payload по сотне тестов.

## Практическая стратегия
Полезная модель:
1. baseline для happy path
2. отдельные reusable наборы для ролей, статусов и тарифов
3. локальные overrides в конкретном тесте
4. edge values для negative и boundary coverage

## Контроль качества данных
Хорошие test data отвечают трём требованиям:
- понятное происхождение
- воспроизводимость
- предсказуемость

Если данные нельзя быстро объяснить и воспроизвести, они будут ломать suite.

## Antipatterns
- mutable shared dataset меняется прямо внутри теста
- случайные данные невозможно повторить
- названия наборов не отражают бизнес-смысл
- один файл с тысячей полей на все случаи жизни

## Что запомнить
- baseline, dynamic и edge data должны быть разделены
- reusable datasets должны быть маленькими и говорящими
- dynamic data нужна там, где есть уникальность и конфликты
- тестовые данные — это часть архитектуры framework
