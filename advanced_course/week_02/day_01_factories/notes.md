# Test Data Engineering, День 1 — Factories

## Зачем это нужно
На небольшом проекте тестовые данные часто создают прямо в тесте. На advanced-уровне это быстро приводит к трём проблемам:
- copy-paste одних и тех же payload
- несогласованные default values
- дорогое обновление схемы сущности

Factory решает это как централизованный конструктор тестовых объектов.

## Что важно понять
Factory — это не обязательно отдельная библиотека. Для большинства QA-задач достаточно функции, которая:
- создаёт валидный объект по умолчанию
- позволяет переопределять отдельные поля
- остаётся предсказуемой и без скрытой случайности

Практическая цель factory:
- быстро собирать happy path
- дешево добавлять edge variations
- поддерживать единый source of truth для формы данных

## Хороший factory
Хороший factory обычно:
- создаёт объект одного доменного типа
- имеет реалистичные default values
- не тянет случайность без явного seed или override
- не прячет критичную бизнес-логику

Пример мышления:
- make_user() создаёт валидного пользователя
- make_order(total=0) позволяет явно проверить boundary case

## Когда factory лучше raw dict
Factory особенно полезен, когда:
- payload используется в десятках тестов
- схема объекта меняется
- нужны небольшие вариации одного и того же объекта
- часть тестов зависит от валидного baseline

Если объект используется один раз и больше нигде не повторяется, raw dict может быть дешевле.

## Antipatterns
Плохие признаки:
- один giant factory для всех сущностей сразу
- дефолты не похожи на реальную доменную модель
- случайные email / id генерируются неуправляемо
- factory сама решает слишком много бизнес-правил

## Что запомнить
- factory сокращает copy-paste и удерживает форму данных в одном месте
- default values должны быть валидными и осмысленными
- variation должна задаваться явно через аргументы
- factory должна быть маленькой и доменно-понятной
