# Test Data Engineering, День 4 — Property-based testing

## Зачем это нужно
Обычные example-based tests проверяют несколько выбранных кейсов. Property-based testing проверяет инварианты на широком множестве входов.

Эта техника особенно полезна для:
- pure functions
- преобразований данных
- нормализации строк и коллекций
- парсеров и валидаторов

## Что важно понять
Property — это правило, которое должно оставаться истинным для большого класса входов.

Хорошие свойства:
- чётко формулируются
- независимы от конкретного одного примера
- проверяют поведение, а не implementation details

Примеры сильных свойств:
- abs(x) >= 0
- sorted(items) возвращает коллекцию той же длины
- normalize_email(x) всегда даёт lowercase string

## Что property-based testing не делает автоматически
Это не просто random testing и не магия.
Нужны:
- корректная стратегия генерации
- понятные свойства
- понимание, где техника уместна, а где нет

## Где особенно полезно
- функции без тяжёлых внешних зависимостей
- правила форматирования и нормализации
- конвертация данных
- логика, где легко выразить инвариант

## Где подходит плохо
- сложный UI-flow
- сценарии с дорогим окружением
- поведение, где oracle трудно формализовать

## Частые ошибки
- вместо свойства писать просто ещё один пример
- выбирать слишком расплывчатое свойство
- пытаться применять подход к тяжёлому end-to-end сценарию

## Что запомнить
- property-based testing проверяет инварианты, а не отдельные кейсы
- техника сильнее всего на чистой логике и данных
- хорошее свойство должно быть проверяемым и однозначным
- инструмент полезен только вместе с ясным мышлением о домене
