# Test Data Engineering, День 2 — Builder pattern

## Зачем это нужно
Factory хорошо работает для простых объектов. Когда сущность имеет много опциональных полей и несколько стадий сборки, factory начинает перегружаться аргументами.

Builder нужен там, где объект удобно собирать пошагово:
- выставить статус
- добавить скидку
- прикрепить теги
- получить финальный payload

## Что важно понять
Builder — это способ собирать объект по шагам без giant constructor.

Типичный признак, что пора переходить к builder:
- аргументов слишком много
- часть конфигурации используется редко
- нужно читать сценарий сверху вниз как pipeline

## Chainable API
Builder чаще всего возвращает self, чтобы можно было писать цепочки:
- OrderBuilder().with_total(100).with_status('paid').build()

Это полезно, когда тест должен быстро показать variation сценария, а не тонуть в большом dict literal.

## Где не нужен builder
Builder не нужен, если:
- объект маленький
- variation мало
- обычной factory достаточно

Builder хорош как advanced-инструмент, но не как default для всего.

## Antipatterns
- builder хранит случайное состояние между тестами
- build() возвращает mutable shared object
- у builder нет валидного baseline состояния
- цепочки становятся длиннее самого теста

## Что запомнить
- builder полезен для сложной и поэтапной сборки данных
- baseline состояние должно быть валидным
- chainable методы должны быть читаемыми
- build() должен возвращать новый и понятный объект
