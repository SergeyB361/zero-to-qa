# Test Data Engineering, День 5 — Edge cases

## Зачем это нужно
Большинство серьёзных дефектов лежит не в happy path, а рядом с границами и нестандартными входами.

Edge cases помогают системно искать:
- пустые значения
- минимумы и максимумы
- значения сразу по обе стороны границы
- редкие, но допустимые комбинации

## Что важно понять
Edge case — это не любой негативный сценарий.
Обычно это значение рядом с boundary или в редкой, но реалистичной зоне поведения.

Практическая модель:
- минимум
- минимум + 1
- типичное валидное значение
- максимум - 1
- максимум
- максимум + 1

## Где искать edges
- длина строк
- суммы, количества, лимиты
- даты и интервалы
- статусы рядом с допустимым переходом
- пустые и почти пустые коллекции

## Частые ошибки
- проверять только одно граничное значение
- забывать значение по другую сторону границы
- путать edge case с полностью произвольной экзотикой

## Что запомнить
- edges почти всегда проверяются около бизнес-лимитов
- границы надо смотреть с двух сторон
- редкий случай должен быть объясним бизнес-правилом
- хороший edge набор компактный, но покрывает критические зоны
